## Findings

### ✅ Comportamento estrutural do sistema é robusto
- **Zero erros HTTP** em 176 310 requests neste run (e total > 437k somando os 3 runs)
- Spike sob 150 VUs absorvido sem dropar conexões em todos os runs
- Detecção lógica preservada (TP/TN/FP/FN consistentes com histórico)
- RSS continua plano e baixo (~17–21 MiB / 160 MB nas APIs)

### ⚠️ Memória extra continua ociosa
- Working set 12–13 % do `mem_limit` — ~140 MB por instância sobrando
- Esta 2ª rodada confirma o achado da 1ª: subir o limite não é exercitado
- Realocação é *headroom* defensivo, não otimização

### ⚠️ Latência depende fortemente do host (Windows + WSL2)
- Throughput baseline variou 642 → 1 358 req/s entre 2 runs idênticos da app
- p99 official variou 140 → 654 → 915 ms sem mudança de código
- **Conclusão:** números absolutos deste ambiente não são úteis pra estimar performance no juiz Linux nativo

---

## Recomendações

1. **Aceitar variância e medir tendências, não valores absolutos** — a diferença entre runs aqui é maior que qualquer otimização razoável de hot path.

2. **Configuração permanece adequada** — `mem_limit: 160M` por API mantém envelope em 350 MB com sobra confortável. Não há justificativa para reverter pra 140 MB.

3. **Foco em sinais binários robustos** que se mantiveram em 100 % dos runs:
   - Sem HTTP errors
   - Sem dropping de conexão sob spike
   - Detecção lógica idêntica
   - RSS dentro do envelope

4. **Para validação confiável de p99**, o juiz oficial Linux nativo (ou um VPS dedicado sem outros containers) é a única referência válida. Windows + WSL2 com containers vizinhos não serve como gate de aceitação.
