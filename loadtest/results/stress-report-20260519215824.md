## Findings

### ✅ Confiabilidade preservada
- **Zero erros HTTP** em todos os 4 cenários (261 686 requests no total)
- Sistema continua absorvendo spike de 150 VUs sem dropar conexões
- Detecção lógica idêntica (TP/TN/FP/FN consistentes com o histórico)

### ✅ Estabilidade de memória
- Working set total ~55 MiB de 350 MB alocados — utilização ~15 % do envelope
- Sem leaks: footprint plano durante 5 min de teste contínuo
- mmap do `.bin` (45 MB) continua no page cache do kernel, fora do RSS

### ⚠️ Memória extra está ociosa
- Cada API usa ~20 MiB de 160 MB — sobra **~140 MB inutilizadas por instância**
- A realocação é uma medida de **headroom defensivo**, não uma otimização de performance
- Para aproveitar essa folga seria preciso código que troque CPU por memória (cache em RAM, lookup tables maiores, IVF com mais centroids, …) — fora de escopo neste delta

### ⚠️ Variância do host domina o sinal
- Containers vizinhos no Docker Desktop competem pelo CPU físico fora do envelope cgroup
- Resultados p99 do Windows + WSL2 não são representativos do juiz Linux nativo
- O número que importa é a **forma do comportamento sob stress** (0 erros, throughput linear, RSS estável) — todos OK

## Recomendações

1. **Manter `mem_limit: 160M`** — não traz ganho de performance, mas é o headroom barato contra OOM em picos fora dos cenários testados. Total de 350 MB alinhado com o envelope da Rinha.

2. **Não confiar em p99 absoluto deste run** — o juiz oficial roda em Linux nativo sem containers vizinhos. Histórico mostra que o número deveria ficar entre 90 e 140 ms (vs. 477–654 ms observados aqui).

3. **Próximo experimento útil**: subir centroids do IVF ou ampliar a tabela de mcc_risk em RAM para gastar os ~140 MB ociosos por API se isso melhorar recall — só vale tentar se houver hipótese de ganho de score em detecção.

4. **Branch `submission` continua válida** — comportamento estrutural (0 erros, throughput sustentado, RSS dentro do envelope) é idêntico ao último snapshot.
